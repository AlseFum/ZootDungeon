# atlas-codegen：JSON → Java 源码生成插件

一个 `buildSrc` 里的 Gradle 插件，负责在构建时读取 `core` 下的 JSON 资源，生成对应的 Java 源文件，再交给 `javac` 一起编译。当前实现把 `*.json` sprite 规格转成 `MobSprite` 子类，后续可以按同样的套路扩到更多生成目标（物品、怪物数据表等）。

本文说明插件的结构、数据流、接入点，以及如何在它基础上新增一种生成任务。

---

## 1. 为什么要做这件事

项目里有一批「内容类 Java 类」其实只是重复套同一个模板——比如每个怪物的 `MobSprite` 子类基本都是：

```java
public class XxxSprite extends MobSprite {
    public XxxSprite() {
        super();
        TextureFilm frames = textureWithFallback("mod:xxx", Assets.Sprites.XXX, 16, 15);
        idle   = new Animation(2,  true);   idle.frames(frames, ...);
        run    = new Animation(10, true);   run.frames(frames,  ...);
        attack = new Animation(15, false);  attack.frames(frames, ...);
        die    = new Animation(10, false);  die.frames(frames,  ...);
        play(idle);
    }
}
```

除了帧号和 fps，几乎没有业务逻辑。把这些写成 JSON，让构建工具批量产出 Java，有几个好处：

- **只描述数据**：JSON 比 Java 更易被美术/策划直接改。
- **统一风格**：生成出来的类格式、命名、注释永远一致。
- **方便再加工**：想一次性给所有 sprite 加上某个字段，只改一个生成器模板即可。
- **不伤运行时**：生成发生在编译期，运行时看到的就是普通 Java 类，不引入反射/热加载的复杂度。

---

## 2. 设计原则

1. **运行时零开销**。不在 `core` 里做 JSON 解析、不在游戏启动时动态拼类。所有成本都落在构建阶段。
2. **`buildSrc` 独立构建**。生成器自己的依赖（Gson 等）不会进入 `core` 的运行时 classpath，保持客户端体积干净。
3. **数据流单向**：`assets/**.json` → 任务生成 `.java` → `javac` 编译。生成文件不进版本库（`.gitignore` 排除），删 JSON 即删类。
4. **失败早**。非法 JSON、未知动画名、帧缺失，直接在 Gradle 阶段报错并停下，避免「编译过了但运行时才崩」。
5. **对 IDE 友好**。生成目录挂到 `sourceSets.main.java.srcDirs`，IDE 能像普通源码一样索引/补全/跳转。

```
ColaDungeon/
├─ buildSrc/                             ← 独立的小 Gradle 构建
│  ├─ build.gradle                       注册插件 + 拉 Gson 依赖
│  └─ src/main/java/com/zootdungeon/codegen/

│     ├─ AtlasCodegenExtension.java      atlasCodegen { } DSL 的数据

   ├─ build.gradle                       应用插件 + 配置 atlasCodegen { }
   ├─ src/main/assets/cola/mob_sprites/  输入：一个 .json = 一个目标类
   │   └─ sample_mob.json

```

## 4. 数据流

```
         sample_mob.json  (手写, 版本库)
                │
                │  generateAtlasSources 任务
                │   ├─ Gson 解析 → SpriteDef POJO
                │   ├─ 严格校验（类名 / 动画名 / 帧 / fps）
                │   └─ 渲染 Java 源码 → 写文件
                ▼
 SampleMobSprite.java  (生成物, gitignore)
                │
                │  sourceSets.main.java.srcDirs 已挂上输出目录
                │  Gradle 自动建立 compileJava → generateAtlasSources 依赖
                ▼
          与 core 其它类一起 javac → 产出 class
                │
                ▼
       运行时 new SampleMobSprite() 即可使用
```

任务执行前会先清空 `outputDirectory`，这样删除一份 JSON，下次构建对应的 `.java` 也会消失，不留僵尸源文件。

## 5. 三个核心文件

### 5.1 `AtlasCodegenExtension`

DSL 数据载体：

```java
public abstract class AtlasCodegenExtension {
    public abstract DirectoryProperty getAssetsRoot();     // 输入目录
    public abstract DirectoryProperty getOutputDirectory();// 输出目录
    public abstract Property<String>  getGeneratedPackage();// 生成类包名
}
```

用 **abstract getter**（Gradle 5+ 推荐写法），由 Gradle 运行时生成实现并自动注入 `ObjectFactory`，省掉手写 `@Inject` 构造器。

对应的用户侧 DSL：

```groovy
atlasCodegen {
    assetsRoot       = file("$projectDir/src/main/assets/cola/mob_sprites")
    outputDirectory  = file("$projectDir/src/autoGenerated/java")
    generatedPackage = 'com.zootdungeon.generated.sprites'
}
```

### 5.2 `GenerateAtlasSourcesTask`

这是 `DefaultTask` 的子类，也是整个插件唯一实际干活的地方。关键点：

- **声明输入/输出**
  - `@InputDirectory`：`assetsRoot`
  - `@OutputDirectory`：`outputDirectory`
  - `@Input @Optional`：`generatedPackage`

  这些注解让 Gradle 能做 UP-TO-DATE 判断——JSON 没改时任务会被跳过，不会每次都重生成。

- **POJO + Gson**

  ```java
  static final class SpriteDef {
      String className;
      String textureLabel;
      String fallbackTexture;
      int frameWidth;
      int frameHeight;
      String initialAnimation;
      LinkedHashMap<String, AnimationDef> animations;
  }

  static final class AnimationDef {
      int fps;
      boolean loop;
      int[] frames;
  }
  ```

  Gson 按字段名直接反序列化；用 `LinkedHashMap` 保留 JSON 里声明顺序（生成时我们实际用固定顺序，以确保产物稳定）。

- **严格校验**

  `validate(def, jsonFile)` 里检查：类名是合法 Java 标识符、`frameWidth/frameHeight > 0`、动画 key 在白名单里（`idle / run / attack / operate / zap / die`——对应 `CharSprite` 里已声明的字段）、`fps > 0`、`frames` 非空、`initialAnimation` 指向存在的动画。任何失败都抛 `GradleException` 终止构建。

- **稳定输出**

  按 `SUPPORTED_ANIMATIONS` 固定顺序渲染动画，保证相同 JSON 每次产物字节级一致——对 diff 审查和构建缓存都友好。

- **文件头注释**

  每份生成文件开头都有：

  ```java
  // AUTO-GENERATED by atlas-codegen. DO NOT EDIT.
  // Source : sample_mob.json
  // Created: 2026-04-22T13:48:07Z
  ```

  想知道某个类从哪来、直接看头两行；IDE 里 `Ctrl+Click` 也能一步跳到源 JSON（如果 IDE 支持 URI 注释跳转）。

- **任务开始时清空输出目录**

  `cleanOutputDir(...)`。避免 JSON 重命名或删除后残留旧类。

### 5.3 `AtlasCodegenPlugin`

做三件事：

```java
public void apply(Project project) {
    // 1) 注册扩展
    AtlasCodegenExtension ext = project.getExtensions()
            .create("atlasCodegen", AtlasCodegenExtension.class);

    // 2) 注册任务并把扩展字段 convention() 接到任务
    TaskProvider<GenerateAtlasSourcesTask> task = project.getTasks().register(
            "generateAtlasSources", GenerateAtlasSourcesTask.class, t -> {
                t.setGroup("codegen");
                t.getAssetsRoot()      .convention(ext.getAssetsRoot());
                t.getOutputDirectory() .convention(ext.getOutputDirectory());
                t.getGeneratedPackage().convention(ext.getGeneratedPackage());
            });

    // 3) Java 插件一就位，就把输出目录挂进 main source set
    project.getPlugins().withType(JavaPlugin.class, p -> {
        JavaPluginExtension java = project.getExtensions()
                .getByType(JavaPluginExtension.class);
        SourceSet main = java.getSourceSets().getByName(SourceSet.MAIN_SOURCE_SET_NAME);

        // 用 TaskProvider.flatMap 而不是直接传 File：
        // Gradle 能据此自动建立「compileJava 依赖 generateAtlasSources」的 task graph。
        main.getJava().srcDir(
                task.flatMap(GenerateAtlasSourcesTask::getOutputDirectory));
    });
}
```

**注意**：这里没有任何 `compileJava.dependsOn(...)`。给 `srcDir` 喂 `TaskProvider` 后，Gradle 会自动把下游编译任务的输入与这个任务的输出绑起来，顺序和增量性都是免费的。

## 6. 约定好的输入格式（当前版本）

```json
{
  "className": "SampleMobSprite",
  "textureLabel": "mod:sample_mob",
  "fallbackTexture": "sprites/rat.png",
  "frameWidth": 16,
  "frameHeight": 15,
  "initialAnimation": "idle",
  "animations": {
    "idle":   { "fps": 2,  "loop": true,  "frames": [0, 0, 0, 1] },
    "run":    { "fps": 10, "loop": true,  "frames": [6, 7, 8, 9, 10] },
    "attack": { "fps": 15, "loop": false, "frames": [2, 3, 4, 5, 0] },
    "die":    { "fps": 10, "loop": false, "frames": [11, 12, 13, 14] }
  }
}
```

| 字段 | 含义 |
|---|---|
| `className` | 生成类名（必须以大写字母开头的 Java 标识符） |
| `textureLabel` | 运行时先查 `SpriteRegistry.the(label)`；`null`/空字符串则永远走 fallback |
| `fallbackTexture` | label 未注册时的资源路径，如 `sprites/rat.png` |
| `frameWidth` / `frameHeight` | 单帧像素尺寸 |
| `initialAnimation` | 构造完后立即 `play(...)` 的动画；缺省时若有 `idle` 则用它 |
| `animations` | 动画映射表；key 只能是 `idle/run/attack/operate/zap/die`（与 `CharSprite` 字段对齐） |
| `animations.*.fps` | 每秒帧数（必须 > 0） |
| `animations.*.loop` | 是否循环 |
| `animations.*.frames` | 帧号数组，对应 `TextureFilm` 里的格子顺序 |

## 7. 使用流程

### 7.1 加一个新怪 sprite

1. 在 `core/src/main/assets/cola/mob_sprites/` 放一份 `goblin.json`，按 §6 填字段。
2. 随手 `./gradlew :core:compileJava`（或 IDE 里刷新 Gradle），即可得到 `com.zootdungeon.generated.sprites.GoblinSprite`。
3. 业务代码 `new GoblinSprite()` 直接用。

### 7.2 单独跑一遍生成

```bash
./gradlew :core:generateAtlasSources
```

控制台会列出每个 JSON 对应的目标类：

```
[atlas-codegen] generating 1 sprite class(es) into .../generated/sprites
[atlas-codegen]   sample_mob.json -> com.zootdungeon.generated.sprites.SampleMobSprite
```

### 7.3 强制重新生成

任务声明了 `@InputDirectory` + `@OutputDirectory`，Gradle 会缓存。如果你只是改了生成器模板（Java 代码里改 `renderJava`），输入没变，任务会被 UP-TO-DATE 跳过。这时用：

```bash
./gradlew :core:generateAtlasSources --rerun-tasks
```

或干脆 `clean` 一下。

## 8. 扩展指南：再加一种生成目标

插件当前只生成 MobSprite 子类，但整体脚手架是通用的。要加新类型（比如从 JSON 生成物品工厂类），两种做法：

### 做法 A：复用现有任务，加一种 POJO + 渲染分支

适合和 sprite 差不多的「一个 JSON 一个类」场景。

1. 在 JSON 里加一个判定字段，比如 `"kind": "item"`。
2. 在 `GenerateAtlasSourcesTask` 里按 `kind` 分发到不同的 `render*` 方法。
3. 校验和模板各自独立，输出仍然落在 `outputDirectory` 下。

优点：只一个任务，构建图简单。
缺点：一个类逐渐变大，违反单一职责。

### 做法 B：再注册一个任务

适合输入完全不同的场景（比如把 `tiles_chel.json` 也生成一份地形索引类）。

1. 新建 `GenerateTileAtlasTask.java`，同样继承 `DefaultTask`。
2. 在 `AtlasCodegenPlugin.apply` 里再 `project.getTasks().register("generateTileAtlas", ...)`。
3. 如果需要独立的 DSL 配置，可以再加一个 `tileAtlas { }` 扩展；或者把 `AtlasCodegenExtension` 改造成包含多个嵌套块：

   ```groovy
   atlasCodegen {
       mobSprites { assetsRoot = ...; outputDirectory = ... }
       tileAtlas  { assetsRoot = ...; outputDirectory = ... }
   }
   ```

4. 同样把新任务的 `outputDirectory` 挂进 `main.java.srcDir`。

做法 B 更干净，推荐用在「输入/输出有显著差异」的情况下。

### 做法 C：拆成独立插件

如果某一类生成将来要在多个子工程复用（比如 `android`、`desktop`、`services` 都要），就把它拆成独立插件 ID（比如 `com.zootdungeon.tile-codegen`），应用插件的子工程各自 `plugins { id '...' }` 即可。`buildSrc` 里可以同时注册多个 plugin id，对应不同的 `Plugin<Project>` 实现。

## 9. 常见坑

- **写了 JSON 但 IDE 里找不到生成类**：`Reimport Gradle Project` 一下。IDE 需要重新读 `sourceSets.main.java.srcDirs` 才能看到新目录。
- **JSON 修改了，生成类却没变**：看 Gradle 日志里 `generateAtlasSources` 是不是 `UP-TO-DATE`。如果是，可能你改的是任务 Java 代码而非 JSON；用 `--rerun-tasks`。
- **`animations` 里的 key 报错**：key 必须在 `idle / run / attack / operate / zap / die` 白名单里。如果真需要新动画字段，先在 `CharSprite` 里加 `protected Animation newKey;`，再把名字加进 `SUPPORTED_ANIMATIONS`。
- **把 `.java` 提交进了仓库**：检查 `.gitignore` 里是否有 `core/src/autoGenerated/`；若历史已被污染，用 `git rm -r --cached core/src/autoGenerated/` 清掉。
- **Gson 严格 JSON 报错**：任务开启了 lenient 模式（能吃无引号键、尾逗号）。但还是建议写标准 JSON，越严格越利于将来换解析器。

## 10. 相关文件一览

| 文件 | 作用 |
|---|---|
| `buildSrc/build.gradle` | 注册插件 ID、拉 Gson 依赖、锁定 Java 21 |
| `buildSrc/src/main/java/.../AtlasCodegenExtension.java` | DSL 字段声明（abstract getter） |
| `buildSrc/src/main/java/.../AtlasCodegenPlugin.java` | 插件入口：注册扩展、注册任务、挂 source set |
| `buildSrc/src/main/java/.../GenerateAtlasSourcesTask.java` | 读 JSON、校验、渲染 Java、写文件 |
| `core/build.gradle` | 应用 `com.zootdungeon.atlas-codegen`、配置 `atlasCodegen { }` |
| `core/src/main/assets/cola/mob_sprites/*.json` | 输入：每份 JSON 对应一个 MobSprite 子类 |
| `core/src/autoGenerated/java/**` | 输出（gitignore）；每次构建清空重建 |
| `.gitignore` | 新增 `core/src/autoGenerated/` 一行，排除生成物 |
