# AI Rule

本文件用于约束 AI 在本仓库中的改动方式。

## 1. 变更范围

- 只修改和当前需求直接相关的文件。
- 不要做无关重构、重命名、格式化全仓库、批量改注释。
- 若发现潜在问题但不在本次需求内，只记录为建议，不直接改。



## 2. 代码风格与结构

- 先读上下文，再写代码；遵循现有命名、抽象和调用风格。
- 优先复用已有能力。
- 避免引入新依赖，除非需求明确要求。
- 不使用Java风格的注释，只用Markdown写注释。
- 所有由AI生成的成员优先public。无特殊情况不得private，不得使用无特殊逻辑的setter/getter
- 内容高度相关的类优先以内部类的形式打包。
- AI生成的文件头部建议维护Table Of Content。不要在函数头加入过多不必要的注释。
- 不必要的getter不要写

## 3. 资源与贴图规则

- 新物品贴图优先通过 `SpriteRegistry.registerItemTexture(...)` 注册后再使用。
- 若贴图仅由单个类使用，优先在该类 `static` 块注册并缓存 image id。
- 不要在实例初始化块里重复注册同一贴图，避免重复占位。
- 资源路径统一使用 `core/src/main/assets/` 下实际存在的文件。不需要考虑文件是否真实存在。如果报错表明不存在，提醒用户。

## 4. Skill 文档规则

- 完整阅读某个类后，在 `docs/skill/` 目录下创建该类的使用说明文档，文件名只使用类的 base name（不含包路径）。
- 在类的开头声明 `@see docs/skill/XXX.md` 或类似引用，指向对应的 skill 文档。
- 当类功能或使用方法变更时，同步更新 skill 文档，以节省后续 AI 理解的 token 开销。
- 内部类直接写在外部类的 skill 文档中，不单独创建文件。
- Skill 文档应包含：类的用法、关键 public 方法说明、典型使用场景、注意事项。

## 5. 提交前自检

- 至少做一次编译或静态检查（若环境允许）。
- 检查是否引入未使用 import、明显空指针风险、越界访问。
- 确认没有误改无关文件（尤其是 `build/`、生成产物、临时文件）。
